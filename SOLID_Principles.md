# 🛠️ 객체 지향 설계 5대 원칙 (SOLID)

프로젝트의 유지보수성과 확장성을 높이기 위한 필수 코딩 규칙 5가지입니다.

---

## 1. SRP: 단일 책임 원칙 (Single Responsibility Principle)

> "하나의 클래스는 단 하나의 책임만 가져야 한다."
> 
- **설명:** 클래스가 변경되어야 하는 이유는 오직 하나뿐이어야 합니다. 하나의 클래스가 너무 많은 일을 하면 버그가 생겼을 때 수정하기 어렵습니다.
- **핵심:** 기능을 조각조각 잘 분리하기.

## 2. OCP: 개방-폐쇄 원칙 (Open-Closed Principle)

> "확장에는 열려 있어야 하고, 수정에는 닫혀 있어야 한다."
> 
- **설명:** 기존의 코드를 변경하지 않으면서도 새로운 기능을 추가할 수 있도록 설계해야 합니다. 보통 **인터페이스(Interface)**를 활용해 구현합니다.
- **핵심:** 기능 추가 시 기존 코드 건드리지 않기.

## 3. LSP: 리스코프 치환 원칙 (Liskov Substitution Principle)

> "자식 클래스는 언제나 부모 클래스를 대체할 수 있어야 한다."
> 
- **설명:** 부모 클래스 자리에 자식 클래스를 넣어도 프로그램이 의도대로 정확하게 동작해야 합니다. 부모의 규약을 위반하는 자식 클래스를 만들면 안 됩니다.
- **핵심:** 상속 관계에서 부모의 기능을 망가뜨리지 않기.

## 4. ISP: 인터페이스 분리 원칙 (Interface Segregation Principle)

> "자신이 사용하지 않는 메서드에 의존하지 않아야 한다."
> 
- **설명:** 거대한 인터페이스 하나보다는, 클라이언트를 위해 잘게 쪼개진 구체적인 인터페이스 여러 개를 만드는 것이 좋습니다.
- **핵심:** 뚱뚱한 인터페이스 대신 날씬한 인터페이스 여러 개 만들기.

## 5. DIP: 의존역전 원칙 (Dependency Inversion Principle)

> "구체적인 것에 의존하지 말고, 추상적인 것에 의존하라."
> 
- **설명:** 고수준 모듈(비즈니스 로직)이 저수준 모듈(구현체)에 직접 의존하면 안 됩니다. 둘 다 추상화(인터페이스)에 의존해야 변화에 유연합니다.
- **핵심:** 구체적인 구현 클래스가 아니라 인터페이스를 바라보게 하기.
